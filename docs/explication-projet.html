Explication du projet IT Monitoring

Document rédigé en français pour comprendre l’architecture et la logique générale du code.
Idée générale
Le projet sert à gérer un parc informatique. Il combine une interface web React, une API FastAPI, une base PostgreSQL et un agent local capable de remonter des informations machines.

1. Vue d’ensemble

L’application est organisée en trois parties principales. Le frontend React affiche les pages, tableaux et formulaires. Le backend FastAPI reçoit les requêtes HTTP, valide les données et parle à la base PostgreSQL. Enfin, la base stocke les utilisateurs, lieux, équipements et logs de maintenance.

Le flux principal est le suivant : le navigateur appelle le frontend, le frontend envoie des requêtes au backend, puis le backend lit ou écrit dans la base de données.

2. Backend FastAPI

Le fichier backend/main.py est le point d’entrée principal de l’API. Il crée l’application FastAPI, configure le CORS pour autoriser le frontend local, puis déclare les schémas de données et les routes HTTP.

Les routes principales couvrent :

  • les utilisateurs ;
  • les lieux ;
  • les types de matériel ;
  • les équipements ;
  • les statistiques pour le tableau de bord ;
  • les logs de maintenance.

Quand le frontend demande une liste d’équipements, le backend interroge la base, charge aussi les relations utiles comme le type et le propriétaire, puis renvoie une réponse déjà pratique à afficher.

3. Modèles de base de données

Le fichier backend/db_models.py décrit les tables via SQLAlchemy. Il représente les objets métier du projet.

  • Location : un lieu physique, par exemple un bureau ou un bâtiment.
  • User : une personne à qui du matériel peut être affecté.
  • Asset : un équipement du parc informatique.
  • MaintenanceLog : une intervention ou un historique de maintenance lié à un équipement.

Ces modèles servent de pont entre le code Python et les tables réelles en base de données.

4. Migrations et structure SQL

Le fichier alembic/env.py prépare le contexte de migration Alembic. Il permet d’exécuter les changements de schéma. Le fichier init.sql crée aussi une structure de base et injecte quelques données d’exemple pour démarrer rapidement l’application.

5. Frontend React

Le fichier frontend/src/index.js démarre React dans la page HTML. Ensuite, frontend/src/App.js installe le routeur et distribue les différentes pages selon l’URL.

La navigation latérale se trouve dans frontend/src/components/Sidebar.js. Elle donne accès aux écrans principaux du projet.

6. Tableau de bord

Le composant frontend/src/components/Dashboard.js concentre l’essentiel de la vue générale. Il charge les équipements, les statistiques par type, ouvre les modales d’ajout et de modification, puis gère aussi la suppression d’un équipement avec confirmation.

Il affiche également un graphique circulaire de répartition par type pour donner une vision synthétique du parc.

7. Inventaire et maintenance

Le composant frontend/src/components/Inventaire.js affiche une autre vue du matériel, orientée consultation de l’historique de maintenance. Lorsqu’un utilisateur clique sur le bouton pour voir les détails, le composant frontend/src/components/MaintenanceLogs.js charge les logs associés à l’équipement sélectionné.

Ce composant permet aussi d’ajouter une nouvelle intervention de maintenance puis de recharger la liste pour afficher le résultat immédiatement.

8. Création et modification d’un équipement

Les composants AddAssetModal.js et EditAssetModal.js pilotent les formulaires de création et de modification. Ils récupèrent les données de référence nécessaires, comme les utilisateurs, les lieux et les types d’équipement, puis envoient les données saisies au backend.

9. Pages d’administration

Les pages UsersPage.js et LocationsPage.js sont des écrans simples d’administration. Elles suivent le même schéma : charger la liste existante, afficher un formulaire, envoyer les nouvelles données au backend, puis rafraîchir l’affichage.

10. Gestion des licences

Le composant Logiciels.js fonctionne actuellement comme une maquette. Les données sont statiques dans le code. Le composant calcule simplement si une licence est valide, bientôt expirée ou déjà expirée selon la date d’expiration.

11. Agent de collecte local

Le fichier agent_collecte.py est prévu pour tourner sur une machine cliente. Il récupère des informations système comme le nom de la machine, l’OS, l’adresse MAC, le numéro de série, le modèle et les logiciels installés. Ensuite, il envoie ces informations au backend via une requête HTTP.

Cet agent sert donc à automatiser une partie de l’inventaire matériel et logiciel.

12. Déploiement

Le fichier docker-compose.yml orchestre les services db, backend et frontend. Le script setup.sh automatise la construction des images Docker, le démarrage des conteneurs et l’initialisation SQL de la base.

Les fichiers Dockerfile du backend et du frontend expliquent comment construire les images nécessaires au fonctionnement de l’application.

13. Point de vigilance

En lisant le projet, on remarque quelques incohérences probables entre certains noms de champs du frontend et ceux du backend, par exemple entre numero_serie et serial_number, ou entre statut et status. Cela n’empêche pas la compréhension globale, mais c’est un point à surveiller si tu veux rendre tout le projet parfaitement cohérent.

14. Conclusion

En résumé, ce projet est une application de gestion de parc informatique avec une architecture web classique : une interface React, une API FastAPI, une base PostgreSQL et un agent de collecte local. Le frontend affiche et édite les données, le backend applique la logique métier, et la base conserve l’état durable du parc.
